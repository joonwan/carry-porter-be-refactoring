# Carry Porter Backend Refactoring

## 이 레포를 만든 이유

이 레포는 기존 `Carry Porter` 프로젝트의 원본 백엔드 구현을 그대로 대체하기 위한 저장소가 아니라, 프로젝트 종료 후 회고를 바탕으로 이벤트 흐름과 배차 구조를 다시 검토하고 더 단순한 구조로 재설계하기 위한 백엔드 레포지토리입니다.

기존 프로젝트에서는 이벤트 기반 후속 처리와 Redis 기반 배차 구조를 통해 실시간 로봇 배차 흐름을 구현했습니다. 다만 프로젝트를 마친 뒤 다시 구조를 돌아보며, 일부 이벤트 흐름은 책임이 분산되어 있었고 배차 로직 역시 현재 규모 대비 복잡도가 높다고 판단했습니다.

이 레포에서는 이러한 회고를 바탕으로 핵심 유스케이스를 다시 정리하고, 이벤트 경계와 배차 책임을 더 명확하게 나누는 방향으로 구조를 재설계합니다.

## 내가 맡은 역할

기존 `Carry Porter` 프로젝트에서는 백엔드 개발자로 참여해 시스템 아키텍처 설계, 이벤트 기반 후속 처리 구조 설계, Redis 기반 배차 시스템 구축을 담당했습니다.

- **시스템 아키텍처 설계**
  - 사용자 요청 처리, 로봇 제어, 실시간 알림 흐름을 분리해 각 구성 요소의 역할이 명확히 드러나는 구조를 설계했습니다.
  - Spring Boot 서버, MQTT 브로커, Jetson 기반 로봇이 각자의 책임에 따라 동작하도록 구성했습니다.

- **이벤트 기반 후속 처리 구조 설계**
  - 사용자 호출 이후 이어지는 배차, 제어, 알림 흐름을 이벤트 기반으로 분리해 요청 처리와 후속 작업의 경계를 나누었습니다.
  - 이를 통해 후속 처리 단계의 변경이 전체 요청 흐름에 직접 영향을 주지 않도록 구성했습니다.

- **Redis 기반 배차 시스템 구축**
  - 가용 로봇 목록과 상태 정보를 Redis에서 관리하는 배차 구조를 구현했습니다.
  - 배차 시점에는 Redis와 Lua Script를 이용해 로봇 선택과 상태 변경을 원자적으로 처리하고, 실패 시 복구 흐름까지 함께 설계했습니다.
